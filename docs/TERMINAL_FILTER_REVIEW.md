## Executive Summary

This document describes the final solution for handling terminal query responses in myworktree. After extensive testing, we found that xterm.js **does send** query responses through `term.onData()`, contradicting earlier assumptions.

### Solution Status: ✅ Working

| Component | Location | Status | Purpose |
|-----------|----------|--------|---------|
| Backend `redact.go` | Log output | **DISABLED** | Protect TUI programs |
| Frontend `term.onData()` | Input path | **ENABLED** | Filter xterm.js responses |
| Frontend state machine | Focus check | **ENABLED** | Prevent early focus |
| Frontend `loadLog()` | Log loading | **ENABLED** | Filter historical logs |
| Frontend WebSocket | Real-time data | **ENABLED** | Filter real-time data |

## 1. Problem Analysis

### 1.1 Original Issue

After merging develop branch, two problems appeared:
1. **TUI display corruption** - Terminal programs (opencode, etc.) displayed incorrectly
2. **Anomalous strings on instance switch** - Strings like `11;rgb:...`, `1;2c`, `;1R` appeared when switching instances

### 1.2 Root Causes

#### Issue 1: TUI Corruption

**Cause**: Backend `controlSeqRemnant` regex in `redact.go` filtered mouse events and cursor position reports from TUI program output.

**Example**: `35;107;1M` (mouse event) was removed, corrupting TUI state.

**Solution**: Disable backend filtering to preserve TUI control sequences.

#### Issue 2: Anomalous Strings

**Cause**: When xterm.js gains focus, it sends terminal query sequences (OSC 11, DA, etc.). The responses are generated by xterm.js and sent through `term.onData()` to PTY.

**Example Flow**:
```
User switches toinstance → xterm.js gets focus → sends OSC 11 query
xterm.js generates response → calls onData callback → sendsto PTY
PTY/shell receives response → displays as garbage text
```

**Key Discovery**: The earlier assumption that "xterm.js responses don't go through onData" was **incorrect**. Testing confirmed that responses ARE sent through `term.onData()`.

## 3. Implementation Details

### 3.1 Frontend Input Filter

**Location**: `internal/ui/static/index.html` - `term.onData()` callback

```javascript
term.onData(data => {// Filter xterm.js auto-generated query responses
    if (isTerminalQueryResponse(data)) {
        // console.debug('Filtered terminal response:', data.length, 'bytes');
        return;
    }
    // ... send toWebSocket
});
```

**Regex Patterns**:

```javascript
function isTerminalQueryResponse(data) {
    if (!data || data.length < MIN_RESPONSE_LENGTH) return false;
    
    // OSC responses: ]11;rgb:..., ]4;palette;rgb:...
    if (data.charCodeAt(0) !== 0x1b) {
        // No ESC - check for partial responses
        if (/^;?\d+(;\d+)?R$/i.test(data)) return true;  // ;1R, 1R
        if (/^\?\d+(;\d+)*\$y$/i.test(data)) return true; // ?2027;0$y
        if (/^rgb:/i.test(data)) return true;           // rgb:...
        return false;
    }
    
    // Has ESC prefix
    if (data[1] === ']' || data[1] === 'P') return true;    // OSC/DCS
    if (data[1] === '[' && data.length > 2 && data[2] === '?') {
        if (/\$\d*y$/.test(data)) return true;  // ?2027;0$y
        if (/\?[\d;]+c$/.test(data)) return true; // ?1;2c
    }
    if (data[1] === '[' && /\[\d+;\d+R$/i.test(data)) return true;
    
    return false;
}
```

### 3.2 Frontend Log Filter

**Location**: `loadLog()`, `loadLogSince()`, WebSocket `onmessage`

```javascript
// Filter function applied to all log content
function filterResponses(text) {
    // OSC 11 color response
    text = text.replace(/\x1b?\]11;[^\x07\x1b]*(\x07|\x1b\\|($|[\r\n]))/g, '');
    // OSC 4 palette
    text = text.replace(/\x1b?\]4;[^\x07\x1b]*(\x07|\x1b\\|($|[\r\n]))/g, '');
    // OSC rgb format
    text = text.replace(/\x1b?\](\d+;)?rgb:[^\x07\x1b\r\n]+/g, '');
    // DEC Private Mode Report
    text = text.replace(/\x1b?\[\?[\d;]*\$y/g, '');
    // Device Attributes
    text = text.replace(/\x1b?\[\?[\d;]+c/g, '');
    // Cursor Position Report
    text = text.replace(/\x1b?\[;?\d+(;\d+)?R/g, '');
    // Partial OSC (missing ESC)
    text = text.replace(/;?\d+(;\d+)*;rgb:[^\r\n]+/g, '');
    // Partial DEC mode (10;16;2$y)
    text = text.replace(/;?\d+(;\d+)+\$y/g, '');
    // Standalone DEC mode (2027;0$y)
    text = text.replace(/\d+;\d+\$y/g, '');
    // Partial cursor report
    text = text.replace(/;?\d+(;\d+)*R(?!\w)/g, '');
    // Standalone Device Attributes (1;2c)
    text = text.replace(/\d+;\d+c(?!\w)/g, '');
    return text;
}
```

### 3.3 Backend Filter - DISABLED

**Location**: `internal/redact/redact.go`

```go
func Text(s string) string {
    s = skPattern.ReplaceAllString(s, "sk-REDACTED")
    s = bearerPattern.ReplaceAllString(s, "Bearer REDACTED")
    // DISABLED: Breaks TUI programs
    // s = controlSeqRemnant.ReplaceAllString(s, "")
    return s
}
```

**Why Disabled**: TUI programs send legitimate control sequences for cursor positioning, mouse events, etc. Backend filtering removes these, corrupting the display.

### 3.4 State Machine

**Location**: `internal/ui/static/index.html` - `connectTTY()`, `focusTerminalIfPossible()`

```javascript
// Prevent focus before WebSocket is ready
function focusTerminalIfPossible() {
    if (!term) return;
    if (!windowHasFocus) return;
    if (openDialogs.size > 0) return;
    if (!state.activeInst) return;
    if (ttyState !== 'READY') return;  // CRITICAL: Only focus when ready
    // ...
    term.focus();
}
```

**Purpose**: Prevent xterm.js from sending query sequences before WebSocket is ready.

## 5. Key Findings - What Changed

### 5.1 Previous Understanding (WRONG)

> xterm.js responses are written to display buffer via `term.write()`, not sent back through `onData`.

**Reality**: xterm.js DOES send responses through `term.onData()` callback.

### 5.2 Previous Understanding (WRONG)

> `term.onData()` handles user keystrokes only.

**Reality**: `term.onData()` handlesBOTH user keystrokes AND xterm.js responses.

### 5.3 Previous Understanding (CORRECT)

> Backend filtering should only remove residues (without ESC prefix).

**Reality**: This is still true, but we disabled backend filtering entirely because it was too aggressive and broke TUI programs.

## 6. Trade-offs

### 6.1 User Intentional ESC Sequences

**Problem**: If a user intentionally types `ESC[c` to query terminal, it will be filtered.

**Mitigation**: This is rare. Most users don't intentionally send ESC sequences.

**Acceptable**: Yes - the benefit of filtering garbage outweighs this edge case.

### 6.2 False Positives

**Problem**: Some legitimate sequences might be incorrectly filtered.

**Mitigated By**:
- State machine prevents early focus (reduces query frequency)
- Regex patterns are carefully designed to minimize false positives
